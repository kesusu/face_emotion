# 工作方法论与迭代规范

> **创建**: 2026-05-11
> **适用范围**: 所有AI模型开发工作（情绪识别CV、ASR、LLM等）
> **核心原则**: 每次工作必留痕，每阶段必复盘，有策略再动手

---

## 一、工作流程闭环（每次会话必须执行）

```
┌──────────────────────────────────────────────────────────────┐
│                      每次工作闭环                             │
│                                                              │
│  ① 接手 → 读文档/读代码 → 理解现状                            │
│       ↓                                                      │
│  ② 制定策略 → 明确目标 → 列任务清单(优先级)                    │
│       ↓                                                      │
│  ③ 执行 → 小步快跑 → 每30min有可运行中间产物                   │
│       ↓                                                      │
│  ④ 记录 → 更新进度MD → 写实验结果 → 记录踩坑                   │
│       ↓                                                      │
│  ⑤ 复盘 → 哪些做对了/做错了 → 总结经验教训                     │
│       ↓                                                      │
│  ⑥ 迭代 → 基于复盘调整策略 → 回到②                           │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 关键规则
- **不准无脑干活**: 先读历史文档，理解前因后果，再动手
- **不准闷头写代码**: 先出分析文档/策略，再写代码；代码是最后一步
- **增量式开发**: 在已有验证通过的代码上改，禁止从零重写（除非明确要求）
- **小步快跑**: 每30分钟出一个可运行的中间结果，不要憋大招
- **先跑通再扩展**: 任何新功能先最小改动验证可行性，确认通了再完善

## 三、复盘与反思模板（每次工作结束必填）

```markdown
### 本次工作复盘 ({日期})

#### 做对了什么
- [记录做得好的决策、高效的做法、正确的判断]

#### 做错了什么 / 可以改进的
- [记录失误、低效做法、判断失误]
- [重要: 分析根因，不是简单说"下次注意"]

#### 经验教训（可复用）
1. [具体的、可操作的规则，不是空话]
2. [例如: "永远先打印state_dict keys对比，不要假设架构相同"]

#### 关键数据变化
| 指标 | 之前 | 之后 | 变化 |
|------|------|------|------|
| 总准确率 | X% | Y% | +/-Zpp |
| 最弱类 | X% | Y% | ... |

#### 下一步行动（按优先级）
1. [P0] xxx — 预期效果 — 预计时间
2. [P1] xxx — 预期效果 — 预计时间
3. [P2] xxx — 预期效果 — 预计时间
```

## 四、迭代优化机制

### 4.1 每次迭代的输入输出

```
输入:
  - 上次的工作汇报 + 复盘
  - README_INDEX.md (当前状态)
  - 用户的新需求/反馈

输出:
  - 本次的工作汇报 + 复盘
  - 更新的README_INDEX.md
  - 更新的比赛汇报（如有新进展）
  - 可运行的代码产出
```

### 4.2 策略制定流程

1. **现状分析**: 数据说话，不拍脑袋
   - 当前各模型/各指标的准确率
   - 瓶颈在哪（最强/最弱类、最大混淆对）
   - 可用资源（数据量、计算力、时间）

2. **方案设计**: 至少2个备选方案，做ROI对比
   - 每个方案: 投入时间 × 预期收益 × 风险 × 依赖条件
   - 用表格对比，选ROI最高的先做

3. **任务分解**: 拆成可执行的小任务
   - 每个任务有明确的完成标准（不是"调参"而是"跑完grid search并记录最优参数"）
   - 标注依赖关系（哪些必须串行，哪些可并行）

4. **执行纪律**: 严格按优先级，不做优先级外的事
   - P0做完才做P1，不要被新想法带偏
   - 如果发现原策略不对，停下来更新策略再继续

### 4.3 总结提升的四层思考

```
Level 1 (记录):  做了什么、结果如何 — 留痕
Level 2 (分析):  为什么是这个结果 — 找根因
Level 3 (抽象):  可以提炼出什么通用规律 — 经验教训
Level 4 (迭代):  基于规律如何改进下次 — 策略优化
```

**只做到Level 1是记流水账，做到Level 4才是真正的迭代优化。**

## 五、比赛汇报撰写指南

### 5.1 核心原则

1. **展示创新性，不暴露实现细节**: 说出"做了什么"+"为什么有效"，不说"具体怎么做的代码"
2. **强调难度，但不诉苦**: 用数据和事实说明问题的挑战性（如"CNN对悲伤只有4%准确率"），不是"调参很难"
3. **突出系统性思维**: 强调"发现了互补关系→设计了路由架构→量化验证有效"，展示工程思维
4. **数据驱动决策**: 每个设计选择都有数据支撑，不是拍脑袋

### 5.2 汇报结构模板

```markdown
# 项目名称 — 阶段性成果汇报

## 项目概述
[一句话说清: 做什么 + 当前达到什么水平 + 相比baseline提升多少]

## 核心创新点（按重要性排序）
### 创新1: xxx
- **发现的问题**: [数据说明问题的严重性]
- **创新思路**: [高层次的解决思路，不涉及具体代码]
- **技术方案**: [架构/算法层面的描述]
- **效果验证**: [前后对比数据]

### 创新2: xxx
...

## 技术难点与突破
### 难点1: xxx
- **挑战描述**: [为什么难，涉及什么技术挑战]
- **突破方法**: [思路层面的描述]
- **量化成果**: [数据对比]

## 成果数据
[表格/图表展示各指标的提升]

## 下一步规划
[展示项目的迭代计划和预期目标]
```

### 5.3 常见表达技巧

| 想表达的内容 | 推荐说法 | 避免说法 |
|-------------|---------|---------|
| 发现两个模型互补 | "系统性地分析了CNN和SVM的逐类性能差异，发现了完美的互补关系" | "CNN强类好，SVM弱类好" |
| 路由架构设计 | "设计了基于SVM7的智能路由架构，将样本分配给对应领域的专家模型" | "if svm说强类就用cnn，else用级联" |
| 调参过程 | "通过全量网格搜索在42个超参组合中找到了全局最优解" | "改了两个阈值试试" |
| 踩坑修复 | "定位了跨平台特征差异导致的模型兼容性问题，通过构建一致性数据流彻底解决" | "层名写错了，改一下就好了" |
| 系统集成 | "构建了多模型智能融合系统，利用各模型的比较优势实现1+1>2" | "把几个模型放一起跑" |

### 5.4 数据展示技巧

- **用对比表格**: "优化前 vs 优化后" 比 "最终结果" 更有说服力
- **用进度条/柱状图**: 可视化各类的准确率差异
- **标注关键突破**: 在数据中标记最重要的提升点
- **给出竞品对比**: 如果有的话，对比业界标准/已知baseline

## 六、交接规范

### 6.1 接手检查清单

```
□ 读 README_INDEX.md — 了解当前状态和文档结构
□ 读最新的工作汇报 — 了解最近做了什么
□ 读最新的复盘分析 — 了解策略和计划
□ 检查脚本是否能跑 — 确认环境/数据/模型文件完整
□ 理解核心架构 — 如Expert v5的路由逻辑
```

### 6.2 离手检查清单

```
□ 更新工作汇报 — 记录本次所有工作内容和结果
□ 更新复盘分析 — 记录经验教训和下一步计划
□ 更新 README_INDEX.md — 更新当前状态和文件索引
□ 更新比赛汇报（如有新进展）— 保持汇报文档同步
□ 所有产出可复现 — 脚本能跑、数据能查、结果能对
```

## 七、历史经验教训库（持续积累）

### CV模型相关

| # | 教训 | 来源 | 适用场景 |
|---|------|------|---------|
| 1 | 永远先打印state_dict keys对比，不要假设架构相同 | V18 CNN=0%排查 | PyTorch模型加载 |
| 2 | 增量式开发，在已验证通过的代码上改，禁止从零重写 | V19 grid search bug | 所有代码开发 |
| 3 | 不要用CNN输出做路由，CNN会偏向训练集分布丰富的类 | Expert v1-5实验 | 多模型融合 |
| 4 | 不要直接加权融合不同模型的概率，概率尺度不可比 | Expert v4实验(43.7%) | 概率融合 |
| 5 | 先跑通最小可运行版本，再扩展完整功能 | V19 grid search性能bug | 功能开发 |
| 6 | PC端和板端SDK的MediaPipe输出有微妙差异，需用板端数据训练 | V11兼容性问题 | 跨平台部署 |
| 7 | 预计算模式：先把所有模型推理跑完存数组，再循环做阈值比较 | V19 grid search重构 | 性能优化 |
| 8 | SNPE模型scores不是标准置信度，训练集也常<0.02，不能用于人脸判断；改用landmarks空间散布度(xy>0.2) | Expert v5.2实拍调试 | 板端SNPE推理 |
| 9 | 多模型决策的Top2不能只看CNN概率，要用多模型投票选次选 | Expert v5.2 Top2修复 | 集成决策输出 |
| 10 | 弱类分支不能完全忽略CNN，CNN高置信度命中强类时应覆盖级联 | Expert v5.1→v5.2 | 多模型路由架构 |
| 11 | 级联对弱类(中性/悲伤)容易误判，RF投票2:1可修正 | photo_28微微笑→悲伤 | 弱类分支防护 |
| 12 | 实拍测试和模型测试要分开脚本：test_live.py(详细) vs test_model.py(统计) | 测试脚本拆分 | 测试脚本设计 |
| 13 | 微微笑→中性可接受，但→悲伤/惊讶不可接受，需区分"错得离谱"和"差一点" | photo_26/28测试 | 情绪粒度评估 |
