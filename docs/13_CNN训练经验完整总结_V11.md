## 二、V11 最佳配置（直接可用）

### 2.1 超参数总表

```python
# ==================== 核心超参数 — 直接复制使用 ====================
MODEL_TYPE = 'deep'          # DeepEmotionCNN (残差网络, 480K参数)
HEATMAP_SIZE = 128            # 热力图尺寸（必须与新数据一致）

# 损失函数

CLASS_WEIGHTS = 'auto'        # 自动计算逆频率权重

# 优化器 + 调度器 ★ 关键组合
OPTIMIZER = 'AdamW'           # 不是SGD! AdamW+OneCycleLR是V11成功的关键
SCHEDULER = 'OneCycleLR'
ONECYCLE_MAX_LR = LR * 10      # max_lr = 0.01
ONECYCLE_PCT_START = 0.1      # 前10%epoch warm-up
ONECYCLE_ANNEAL = 'cos'       # 余弦退火

# 数据增强
AUGMENT_PROB = 0.8            # 80%概率做增强
FLIP_PROB = 0.5               # 水平翻转概率
ROTATE_RANGE = (-20, 20)      # 旋转范围(度)
ROTATE_PROB = 0.6             # 60%概率旋转
NOISE_STD = 0.02              # 高斯噪声标准差
NOISE_PROB = 0.5              # 50%概率加噪声
BRIGHTNESS_RANGE = (0.85, 1.15)  # 亮度变化

# 数据划分
TRAIN_VAL_SPLIT = 0.2         # 验证集占20%
STRATIFY = True               # 分层抽样(保持类别比例)
RANDOM_STATE = 42             # 随机种子(保证可复现)
OVERSAMPLING = True           # 对训练集做加权过采样

# 梯度裁剪
GRAD_CLIP_NORM = 5.0          # 防止梯度爆炸
```

## 三、训练演进历史（踩过的坑）

### V9 → V10 → V11 对比

| 配置项 | V9 | V10 | **V11★** |
|--------|-----|------|----------|
| 优化器 | SGD+Nesterov | AdamW | **AdamW** |
| 学习率 | 1e-2 | 3e-3 | **1e-3** |
| 调度器 | CosineAnnealing | ReduceLROnPlateau | **OneCycleLR** |
| Epochs | 80 | 150 | **200** |
| Dropout | 0.4 | 0.5 | **0.5** |
| WeightDecay | 1e-4 | 1e-3 | **1e-3** |
| LabelSmoothing | 无 | 无 | **0.05** |
| **ValAcc** | **39.7%** | **40.4%** | **52.1%** |
| BestEpoch | ~75 | 19(过早过拟合!) | **195(持续学习)** |

### 各版本教训

#### V9 (SGD + CosineAnnealing, 80ep) → 39.7%
- SGD在大数据集上收敛慢，80ep不够
- CosineAnnealing没有warm-up阶段，前期lr太高不稳定
- **悲伤仅9%、中性22%** — 弱类别完全没学到

#### V10 (AdamW + ReduceLROnPlateau, 150ep) → 40.4%
- **LR=3e-3对AdamW太高** → epoch 19就过拟合(val_acc开始下降)
- ReduceLROnPlateau是被动响应式调度，等到val_loss不降才减lr，太晚了
- 效果反而比V9还差点

#### V11 (AdamW + OneCycleLR, 200ep) → **52.1% ★**
- **关键改进1**: LR降回1e-3，OneCycleLR的warm-up让前20ep稳定上升
- **关键改进2**: OneCycleLR的余弦退火让模型持续学习到epoch195
- **关键改进3**: LabelSmoothing 0.05 + 更强数据增强缓解过拟合
- 悲伤25.6%、中性52.9% — 大幅提升！

### V11训练曲线特征

```
TrainAcc:  24% → 83% (持续上升, 几乎没平台期)
ValAcc:    16% → 52% (持续上升, epoch195才到峰值!)
ValLoss:   1.92→1.80 (缓慢下降后趋于平稳)

⚠️ Train-Val Gap很大 (~30%), 这是正常的:
  - 过采样导致train_acc虚高
  - 数据增强只对train生效
  - ValAcc才是真实指标
```

## 四、数据格式要求

### 4.1 输入npy文件规格

新热力图必须满足以下格式：

```python
# 文件路径: RAF-DB/train/reports/cnn_train_data_128.npy

import numpy as np
data = np.load('cnn_train_data_128.npy', allow_pickle=True).item()

data.keys() 必须包含:
├── 'heatmaps':   np.ndarray, shape=(N, 3, H, W), dtype=float32, 值域[0,1]
├── 'labels':     np.ndarray, shape=(N,), dtype=int64, 值域[0,6]
├── 'features':   np.ndarray (可选), shape=(N,14), SVM用的
├── 'file_paths': list[str] (可选)
├── 'label_order': list[str] = ['惊讶','恐惧','厌恶','开心','悲伤','愤怒','中性']
├── 'categories': dict (可选)
├── 'total_samples': int
└── 'heatmap_size': int (应为128)
```

### 4.2 热力图通道含义

```
Channel 0: xy位置热力图 — 所有468个关键点的xy投影高斯叠加
Channel 1: z深度加权热力图 — z值凸出面更亮 (z∈[-1,1] → weight∈[0,1])
Channel 2: 区域分组热力图 — 不同区域不同权重:
           眉毛(w=2.0) + 眼睛(w=2.5) + 鼻子(w=1.0) + 嘴巴(w=3.0) + 轮廓(w=0.8)

每个channel独立归一化到[0,1]
高斯核: sigma = GAUSS_SIGMA * size / 10 = 1.5 * 128 / 10 = 19.2像素
```

## 五、模型架构详情 (DeepEmotionCNN)

```
DeepEmotionCNN — 参数量: 480,743

输入: (B, 3, 128, 128) float32 [0,1]

├─ stem: Conv2d(3→24, k3p1) → BN → ReLU → MaxPool(2)       → (B,24,64,64)
│
├─ stage1: Conv(24→32,k3p1) → BN → ReLU → ResBlock(32)     → MaxPool(2) → (B,32,32,32)
│
├─ stage2: Conv(32→48,k3p1) → BN → ReLU → ResBlock(48)     → MaxPool(2) → (B,48,16,16)
│
├─ stage3: Conv(48→64,k3p1) → BN → ReLU → ResBlock(64)     → AvgPool(4×4) → (B,64,4,4)
│
├─ classifier:
│   Flatten → (B,1024)
│   Linear(1024→256) → BN → ReLU → Dropout(0.5)
│   Linear(256→128)  → ReLU → Dropout(0.25)
│   Linear(128→7)    → logits
│
└─ 输出: (B,7) 未归一化的logits
```

### 权重初始化
- Conv2d: Kaiming Normal (fan_out, relu)
- BatchNorm: weight=1, bias=0
- Linear: Xavier Normal, bias=0

## 七、已知坑点 & 注意事项

### 7.1 训练相关

| 坑点 | 说明 | 解决方案 |
|------|------|---------|
| scheduler.step()位置 | OneCycleLR每epoch结束时调用一次，**不需要传参** | 放在验证循环结束后 |
| cw_tensor定义顺序 | criterion用到cw_tensor，必须在criterion之前定义 | 先定义cw_tensor再写criterion |
| Python环境 | 训练需sklearn(`StratifiedShuffleSplit`) | 用 `py -3.8`，不是 `py -3.11` |
| GPU确定性 | RTX 4060 Laptop有fp16噪声 | 强制float32 + deterministic模式 |
| 模型保存命名 | 脚本固定存为 `v9_cnn_deep_best.pth`(命名遗留) | 实际内容取决于训练时的model_type |

### 7.2 数据相关

| 坑点 | 说明 |
|------|------|
| MediaPipe版本差异 | PC和板子mp版本不同 → 热力图分布不同 → **模型不通用** |
| 人脸检测失败率 | PC端~13%(1303/9906)，板子可能更低 → 新数据量可能更多 |
| 数据不平衡 | 恐惧仅288张 vs 中性2635张 → 已通过WeightedRandomSampler处理 |
| 热力图shape必须是(N,3,H,W) | 不要搞成(H,W,3)或(N,H,W) |

### 7.3 如果新数据量变化

```
如果新热力图 > 10000张:
  → BATCH_SIZE 可以尝试 96 或 128 (如果GPU显存够)
  → EPOCHS 可以降到 150 (数据多了收敛更快)
  
如果新热力图 < 7000张:
  → EPOCHS 提高到 250-300 (防止欠拟合)
  → 数据增强可以更强 (AUGMENT_PROB=0.9)
```

## 十、V19/V19g 新增经验 (2026-05-12 更新)

> 以下经验来自V19系列在板端SDK热力图(9906张7类)上的训练。

### 10.1 中性偏执问题 — CNN集成到Expert时的致命陷阱

**现象**: V19f ValAcc=47.9%看起来不错，但集成到板端Expert后整体**暴跌16pp**(64.6%→48.3%)。

**根因**: CNN把太多非中性样本判成中性：
- 厌恶→中性: 53% (过半!)
- 悲伤→中性: 57% (过半!)
- 恐惧→中性: 33%
- 惊讶→中性: 33%

Expert的强类分支依赖CNN做正确预测，CNN说"中性"时，SVM7路由到强类的样本就被错误覆盖了。

**V15(30.6%)为什么没这个问题**: V15对中性识别率仅12%，几乎不预测中性→V15的强类预测更"干净"。

### 10.2 FocalLoss alpha 是解决中性偏执的关键工具

V19g通过在FocalLoss中给中性类降权alpha，成功将非中性→中性误判从56%压到2.2%：

| alpha[中性] | 非中性→中性max | 中性recall | ValAcc | 评价 |
|-------------|---------------|-----------|--------|------|
| 无(V19f) | 56% | 83.6% | 47.9% | 中性偏执严重 |
| 0.5(V19g-a) | 2.2% | 3.7% | 40.2% | ★ 最佳平衡 |
| 0.3(V19g-b) | 0.0% | 0% | 33.2% | 过度: 中性被压死 |
| 0.7(V19g-c) | 0.5% | 1.0% | 32.4% | 温和但效果弱 |

**结论**: alpha[中性]=0.5 是最佳值。**0.3太低会完全压死中性，0.7太高效果弱。**

### 10.3 中性样本截断是有效的辅助手段

V19g-a同时用了两个中性抑制手段：
1. **FocalLoss alpha[中性]=0.5**: 损失函数层面降权
2. **中性样本截断3074→1500**: 数据层面减量，让各类≈800-1600张更均衡

截断后中性仍是最多类(1500 vs 悲伤2141不截断)，但不再是绝对多数(31%→19%)。

### 10.4 Label Smoothing 会加剧中性偏执

Label Smoothing让模型对正确类别"不那么确信"，对于已经倾向中性的CNN来说，这会：
- 把本该属于强类的置信度分散出去
- 中性因为本身是个"模糊"类别，反而受益
- **V19g-a把LS从0.1降到0.05，V19g-b直接关掉LS=0.0**

| LS | 效果 |
|----|------|
| 0.1(V19f) | 中性偏执严重 |
| 0.05(V19g-a) | 中性偏执大幅改善 |
| 0.0(V19g-b) | 中性被完全压死(配合alpha=0.3) |

**经验**: 如果模型有中性偏执，**第一步就降LS**。

### 10.5 Dropout对中性偏执的影响

| dropout | 模型 | 中性偏执 |
|---------|------|---------|
| 0.6 | V19f | 严重(56%) |
| 0.5 | V19g-a | 改善(2.2%) |

Dropout=0.6太强可能导致非中性类欠拟合，模型学不到足够强的非中性特征→退而求其次判中性。
降到0.5让非中性类有更强的学习能力。

### 10.6 ValAcc不是唯一指标 — 集成场景下的正确评价

| 模型 | 独立ValAcc | Expert集成Top1 | 说明 |
|------|-----------|---------------|------|
| V15(30.6%) | 低 | 64.6% | 中性recall仅12%, 但强类"干净" |
| V19f(47.9%) | 高 | 48.3% | 中性偏执污染了Expert路由 |
| V19g-a(40.2%) | 中 | 待验证 | 中性偏执已修复, 预期集成提升 |

**核心教训**: CNN不是独立使用的，它的输出是Expert路由的输入。CNN的"误判方向"比"整体准确率"更重要。一个40%但中性误判2%的CNN，可能比47.9%但中性误判56%的CNN在集成中表现更好。

### 10.7 训练脚本管理教训 — 文件覆盖问题

V19g训练时覆盖了V19f的best checkpoint (`v19_cnn_new.pth`)，导致推送了错误的模型(46.1%@ep29而非47.9%@ep66)。

**教训**:
1. **模型文件名必须包含版本+准确率**: `v19f_cnn_acc48.pth` 而非 `v19_cnn_new.pth`
2. **每次训练前确认输出路径不会覆盖之前的best**
3. **推送前用Python验证模型内部数据**: `ckpt['best_val_acc']`, `ckpt['epoch']` 必须和文件名一致
4. **deploy副本只保存模型权重**，不含优化器(1.9MB vs 5.8MB)

### 10.8 板端文件系统截断问题

板子对推送到含中文/长路径的文件，**会截断文件扩展名**：

| 原始文件名 | 板子上实际 | 问题 |
|-----------|-----------|------|
| `.pth` | `.` | 扩展名变点 |
| `.json` | `.j` | 截断 |
| `.md` (长名) | 无后缀 | 完全丢失 |

**对策**: 板端代码不要硬编码扩展名，用 glob 或前缀匹配加载模型。
